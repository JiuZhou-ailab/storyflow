# Implementation Notes

## 2026-05-27

- Investigating a writing editor selection bubble drift where the formatting and AI prompt BubbleMenus are appended under TipTap's editable DOM inside the scrollable document content. The intended fix is to anchor floating UI to the stable editor shell, listen to the editor content scroll target for repositioning, and hide when the selected reference leaves the clipping boundary.
- Implemented the selection bubble fix at the shared `TiptapMarkdownEditor` / `TiptapBubbleMenus` boundary. The two text-selection menus now use the editor shell as `appendTo`, the `.tiptap-editor-content` element as Floating UI `scrollTarget`, and `hide: true`; rich block and inline math menus are left unchanged for now because the reported drift is selection-text specific and those menus have custom node anchors.
- Revised the selection bubble behavior after live feedback: using Floating UI `scrollTarget` alone still lets TipTap chase the selection during scroll and snap back after debounce. Text selection menus are now dismissed immediately on editor-content scroll and only re-enabled by a subsequent selection update, preventing stale menus from appearing over unrelated manuscript text.
- Reworked the Web UI login page as a conventional responsive auth shell instead of a single centered card. Desktop now separates brand context from the auth form; mobile collapses to one column. The auth logic stays on the existing Neon/email, Feishu, and server-token endpoints.
- Kept server-token login as a fallback method and made dividers conditional. This avoids showing a bare `or` when only one auth method is configured, while preserving deployments that still rely on `CRAFT_SERVER_TOKEN`.
- Revised the Web UI login visual direction back toward the main app chrome: a compact single auth card, system font, app-like oklch foreground/background tokens, purple accent, minimal border/shadow, and no marketing-style brand panel.
- Audited skill loading for global/local behavior before adding writing skills. Craft loads global skills from `~/.agents/skills`, workspace skills from `{workspace}/skills`, and project skills from `{workingDirectory}/.agents/skills` with project > workspace > global precedence.
- Treated the screenshot requirements as global Craft skills, not short-form Method Pack-local skills. Existing global skills cover outline generation, ideation, short-form diagnosis, plot causality, timeline, character, chapter unit, and state ledger; missing global skills are being added under `~/.agents/skills`.
- Consolidated the screenshot coverage to avoid one-function-per-skill duplication: `prose-drafting` covers continuation and outline-to-prose, `prose-revision` covers expansion/polish/de-AI, `story-deconstruction` covers deconstruction and benchmark rewrite, and AI correction is folded into `plot-causality-audit`.
- Chinese-localized all global Craft writing skill metadata and UI prompts under `~/.agents/skills`. Folder slugs remain English as stable IDs, while visible names, descriptions, bodies, and default prompts are Chinese.
- Rechecked duplicate boundaries: overlaps are intentional trigger overlap, not duplicate ownership. Drafting, revision, deconstruction, ideation, outlining, causality, timeline, character, chapter-unit, state-ledger, and short-form diagnosis remain separate responsibilities.
- Noted a live-refresh boundary: global skills are watched directly, but project-level `.agents/skills` outside the workspace are only picked up when `workingDirectory` is passed through explicit reload paths; some watcher callbacks reload without projectRoot.
- Localized the generated short-form Method Pack skills at the source (`short-opening-designer`, `short-golden-three`, `short-draft-chapter`, `short-reviser`) while keeping their English slugs as stable IDs. New short-form workspaces now generate Chinese visible skill names, descriptions, headings, and bodies.
- Migrated existing generated short-form workspace skill copies under `~/.craft-agent/workspaces` to the Chinese templates. Legacy short-article skills were translated in place when they still had the old generated marker, but were not deleted because they may belong to older user workspaces.
- Fixed the post-login project layout path at two separate layers: new Electron windows now reset renderer zoom to `1`, and the shell no longer resolves desktop sidebar/workspace proportions while the container is still in compact/narrow measurement. The writing workspace width is also clamped so the assistant panel keeps its minimum usable width.
- Kept the existing managed `wangsu-default` gateway as the default AI path. `getSetupNeeds()` already treats a managed custom `baseUrl` as keyless-capable, so users who pass client auth should not see AI provider onboarding; if the gateway later requires a service credential, `CRAFT_BUILTIN_LLM_API_KEY` can seed it into the encrypted credential store without committing a key to `config-defaults.json`.
- Kept SDK-fork branch safety intact. Missing Pi/Claude provider anchors still reject on the server; the UI now receives/stores `canBranch` metadata and rewind selection skips assistant messages that cannot be safely forked, avoiding the internal "Pi branch anchor missing" error path for old/incomplete turns.
- Added a broad session-turn watchdog instead of per-tool/provider timeout plumbing. The policy is intentionally generous but bounded: 30 minutes without any agent event is treated as idle timeout, and 12 hours is the absolute hard turn limit.
- Kept timeout policy isolated in `packages/server-core/src/sessions/turn-watchdog.ts` so `SessionManager` only starts the watchdog, marks progress on agent events, and performs one existing `AbortReason.Timeout` cleanup path.
- Timeout cleanup adds a visible error message, calls provider `forceAbort(AbortReason.Timeout)`, and reuses `onProcessingStopped(..., 'timeout')` so queued messages and browser cleanup follow the existing session lifecycle.
- This first pass deliberately does not add separate MCP/tool/permission TTLs. Those would be more precise but would spread timeout policy across many adapters; the current fix is a coarse recovery rail with minimal code pollution.
- Added a Neon-side read view `public.craft_auth_users` on project `default` / database `neondb` so Auth users are visible from the default `public` schema without selecting sensitive Better Auth account fields. The source of truth remains `neon_auth."user"`; the view currently exposes Neon Auth email/password users only, not Feishu OAuth logins handled outside Neon Auth.
- Promoted the bundled `wangsu-default` gateway to the hidden default connection whenever bundled defaults are applied. User-facing rewind/new-session flows should not inherit stale historical custom connections; historical provider-native rewind is blocked when the source session is locked to a different connection than the current hidden default.
- Moved the WindowManager native-window regression test to `window-manager.isolated.ts`. It mocks Electron at module scope, and Bun keeps module mocks process-wide within a test run; isolating it prevents the partial `electron`/`logger` mocks from leaking into BrowserCDP, deep-link, and browser-pane tests.
- Softened long writing-catalog file titles in the shell sidebar by letting compact file items use a two-line clamp with a stable minimum row height. - Wired successful client auth into the managed model gateway credential slot. Neon email/password sign-in stores the verified Neon token for `CRAFT_CLIENT_GATEWAY_LLM_CONNECTION_SLUG` (default `wangsu-default`), while Feishu broker sign-in stores the broker-issued `gatewayToken` and clears that exact managed credential on sign-out.

## 2026-05-26

- Implementing Feishu login at the Web UI auth boundary, not in the Lark messaging bot path: the project already has a browser `/login` flow, while the bot adapter only authenticates platform traffic.
- Keeping password login as a fallback so existing headless deployments using `CRAFT_SERVER_TOKEN` / `CRAFT_WEBUI_PASSWORD` keep working.
- Treating the company-internal rule as a Feishu `tenant_key` allow-list. External Feishu accounts must match an active registration row in the configured Postgres/Neon store.
- Neon MCP did not complete its handshake in this session, so no live Neon project or credential was created. Based on official Neon regions, China-side latency should prefer `aws-ap-southeast-1` (Singapore); the chosen project region cannot be changed after project creation.
- Avoiding a hard dependency on a specific Neon project in source code. Runtime config will use `CRAFT_WEBUI_AUTH_DATABASE_URL`, with schema creation handled lazily by the registration store.
- Added unauthenticated Web UI endpoints under `/api/auth/feishu/*`: `config`, `start`, and `callback`. OAuth `state` plus PKCE protects the browser callback before a Craft session cookie exists.
- Added `CRAFT_WEBUI_FEISHU_APP_ID`, `CRAFT_WEBUI_FEISHU_APP_SECRET`, `CRAFT_WEBUI_FEISHU_REDIRECT_URI`, `CRAFT_WEBUI_FEISHU_SCOPE`, `CRAFT_WEBUI_FEISHU_INTERNAL_TENANT_KEYS`, and `CRAFT_WEBUI_AUTH_DATABASE_URL`. Feishu login is enabled only when app ID and secret are both present.
- The Postgres registration table is `webui_feishu_users`. Manual external registration should set `status = 'active'` on a matching `open_id`, `union_id`, `user_id`, `email`, or `enterprise_email`; unregistered external login attempts are recorded as `pending` when the database URL is configured.
- Retested Neon MCP after OAuth login. The available `default` project is in `aws-ap-southeast-1` and runs PostgreSQL 17.10. A temporary branch `codex-feishu-auth-test-20260526` was created for validation, the `webui_feishu_users` DDL and registration lookup behavior were verified there, and the temporary branch was deleted afterward.
- Starting a minimal Neon Auth bridge for email sign-up/sign-in. The auth boundary stays at Web UI session issuance: Neon Auth verifies user identity, then the server issues the existing `craft_session` cookie so model access still goes through the current gateway.
- Deferring token grants, quotas, and billing tables. The first stable target is one unified user-facing auth surface; accounting can be added later behind the same session subject without changing the login contract.
- Avoided adding a browser SDK dependency after Bun workspace install attempted to resolve the local WebUI package from npm. The implemented path uses server-side Neon Auth REST calls for email sign-in/sign-up, then verifies the returned JWT with JWKS before issuing the local session cookie.
- Provisioned Neon Auth on the Neon `default` project main branch (`young-cherry-15654428` / `br-square-feather-a1kv8pnz`) for database `neondb`. Email/password sign-up and sign-in are enabled without mandatory email verification for the first stable auth pass; local trusted origins are limited to localhost/127.0.0.1 ports 9100 and 3100.
- Verified the branch-specific Neon Auth OpenAPI schema and corrected the server bridge to call `/sign-in/email` and `/sign-up/email` relative to the Neon Auth base URL, not `/auth/sign-in/email`.
- Added username login as a server-side mapping to Neon Auth email/password: `zjding` becomes `zjding@users.craft.invalid` when `CRAFT_WEBUI_NEON_AUTH_USERNAME_EMAIL_DOMAIN=users.craft.invalid` is set. This keeps Neon Auth as the only identity store while preserving a short username login for the owner account.
- Added `CRAFT_WEBUI_PASSWORD_AUTH_ENABLED=false` so deployments can require account auth and hide/reject the old server-token Web UI login without changing the RPC server token contract.
- Neon Auth `/sign-in/email` returns a Better Auth opaque session token in the top-level `token` field. The Web UI bridge must treat that as a session token, read the auth cookie, call `/token`, and verify the returned JWT before issuing `craft_session`.
- Clarified that the earlier Neon Auth Web UI bridge only protects the browser/HTTP Web UI path. `bun run electron:dev` starts the Electron renderer directly and does not visit that Web UI login page.
- Added Electron client auth as a separate startup gate: the main process owns Neon Auth sign-in state, preload exposes only auth IPC for login, and the renderer lazy-loads `App` only after the account is authenticated.
- Reused the existing Neon Auth config for the Electron client to avoid a second identity store. The local `.env` now sets `CRAFT_CLIENT_AUTH_REQUIRED=true`; client-specific Neon env vars can override the Web UI Neon values later if needed.
- Kept Electron auth state process-local for this first stable pass. There is no persisted refresh token and no billing/quota layer yet; restarting the app requires signing in again.
- Preload now blocks normal RPC invoke calls before client auth succeeds. This is a client startup/user-flow gate for the desktop app, not a hard security sandbox against someone modifying local app code.
- Fixed Electron email/password login by always passing a trusted Origin to Neon Auth. The default is `http://localhost:9100`, configurable with `CRAFT_CLIENT_NEON_AUTH_ORIGIN`; the value must stay in Neon Auth trusted origins.
- Added Electron Feishu OAuth login to the same client auth gate. The main process opens Feishu in the system browser, listens for `http://localhost:6477/callback`, and sends the code plus PKCE verifier to the configured auth broker.
- Electron Feishu client config is intentionally public-client-only. Access policy and Feishu app secret live on the broker/server; for Electron, register `http://localhost:6477/callback` in the Feishu Open Platform app.
- Electron Feishu login no longer reads client-side allow-all or tenant allow-list env vars. Those decisions belong to the broker so the distributed app cannot be changed into a different access policy by local env edits.
- Added a Feishu login timeout and cancel path for Electron. If Feishu rejects or never calls back to `http://localhost:6477/callback`, the auth gate no longer spins forever and the loopback callback server is closed before retry.
